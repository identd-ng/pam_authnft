Contributing to pam_authnft
===========================

pam_authnft is a PAM session module that binds nftables rules to a user
session via its cgroupv2 path. It runs inside the PAM process, under a
seccomp-BPF allowlist, and talks to systemd over D-Bus and to the kernel
via libnftables. Changes to any of those three surfaces have narrow
tolerances — read this file before opening a pull request.


Build
-----

    make               # release
    make debug         # -DDEBUG -g, stderr tracing
    make test          # unit tests, no root
    sudo make test-integration
                       # pamtester + valgrind, requires root

pkg-config probes six libraries: libnftables, libseccomp, libsystemd,
pam, libcap, audit. The hardening flag set in the Makefile is load-bearing —
stage 6 of the test suite greps `checksec` output for Full RELRO.

Debug bypass (env or PAM arg) disables seccomp only:

    AUTHNFT_NO_SANDBOX=1


Layout
------

    src/pam_entry.c    PAM entry points; lifecycle ordering
    src/sandbox.c      seccomp-BPF allowlist
    src/bus_handler.c  sd-bus StartTransientUnit + cgroup path validation
    src/nft_handler.c  fragment validation, nft transaction, cleanup
    include/authnft.h  public contract, constants
    tests/             unit + pamtester integration harness
    data/authnft.slice parent slice unit
    examples/          fragment generator
    pam_authnft.map    exported symbol whitelist (exactly two entries)


Invariants to preserve
----------------------

1. Exported symbols are `pam_sm_open_session` and `pam_sm_close_session`
   only. Adding a new PAM hook requires editing `pam_authnft.map`.

2. Lifecycle order in `pam_sm_open_session` is fixed: validate user →
   root bypass → parse `PAM_RHOST` → `sandbox_apply` → `bus_handler_start`
   → construct `cg_path` from scope_unit → `pam_set_data` →
   `nft_handler_setup`.

3. Session state is persisted via `pam_set_data` keyed `"authnft_cg_id"`.
   The stored value is `authnft_session_t` (cg_path + scope_unit +
   normalized remote_ip + optional claims_tag + correlation_id); the
   key name predates the
   struct and is kept for compatibility. `close_session` reads the
   struct back to delete the exact element `open_session` inserted
   (including which set it went into) and to reuse the correlation_id
   in the close-side audit event. Do not re-resolve the cgroup from
   the PID at close — the transient scope may already be gone.

4. Error asymmetry is intentional: open_session is strict, close_session
   is best-effort. Cleanup failures log a warning and return
   PAM_SUCCESS so a session can always unwind.

   Failed open_session paths roll back any partial state they created:
   `nft_handler_setup` calls `nft_partial_cleanup` on every error path
   that fires after call 1 succeeded (removes per-session chain + 3
   sets, plus the jump rule when its handle was captured), and
   `pam_sm_open_session` calls `bus_handler_stop` after any failure
   that occurred after `bus_handler_start` succeeded. The result is
   that a failed open leaves no nft state, no transient scope, and no
   session-identity JSON — a clean retry is always safe. Integration
   test 10.14 guards both rollbacks. Documented edge case: if call 2
   commits the jump rule but the kernel-handle parse fails, the rule
   leaks because nftables has no delete-by-name primitive.

5. Fragment validation is strict: `stat()` must succeed, `st_uid == 0`,
   world-writable bit must be clear. Failure is a user-visible
   PAM_AUTH_ERR via `pam_error()`. This is the trust boundary — treat
   changes as security changes.

6. The nft transaction is three libnftables calls:
       (a) add table + shared filter chain + ct state rule +
           per-session chain + 3 per-session sets + element
       (b) add jump rule with ECHO|HANDLE for handle capture
       (c) read fragment, substitute placeholders, execute
   The element is inserted before the fragment so fragment rules can
   reference the per-session sets via placeholders.

7. Each session creates three per-session sets:
       session_<user>_<pid>_v4   typeof socket cgroupv2 level 2 . ip saddr;  flags timeout
       session_<user>_<pid>_v6   typeof socket cgroupv2 level 2 . ip6 saddr; flags timeout
       session_<user>_<pid>_cg   typeof socket cgroupv2 level 2;             flags timeout
   A session element goes into exactly one of these, selected by
   PAM_RHOST normalization and the configured rhost_policy. A jump
   rule in the shared filter chain dispatches to the per-session
   chain `session_<user>_<pid>`. Cleanup deletes by handle.

8. Chain priority is `filter - 1` so pam_authnft runs before the default
   filter chain.


Concurrency
-----------

Multiple pam_authnft invocations can run in parallel (e.g., simultaneous
sshd sessions). The `add table` and shared `add chain filter` statements
are byte-identical and idempotent. Per-session chains and sets carry
unique names (session_<user>_<pid>_*) and do not collide across
sessions. Each session's jump rule and fragment rules are isolated in
the per-session chain; there is no cross-session critical section.

Fragments may use nftables `include` directives (integration test 10.8
covers transitive resolution). Invariant #5's ownership check is
applied only to the fragment itself — files pulled in transitively are
NOT re-validated. pam_authnft logs a LOG_INFO on open when a fragment
contains an `include` directive; administrators remain responsible for
ensuring included files are root-owned and not world-writable.


Seccomp allowlist
-----------------

The allowlist in `src/sandbox.c` is derived empirically. The procedure:

    1. Build with `make debug`.
    2. Run `strace -f` across a complete `open_session` + `close_session`
       cycle under pamtester.
    3. Include only syscalls observed after `sandbox_apply()` returns.

`execve(2)` appears in the trace but comes from pamtester startup before
`dlopen()` loads the module — it is deliberately excluded. If you add
code that reaches a new syscall from any of the three handlers, you must
add the syscall here or the module is killed with SIGSYS at runtime.
Default action is `SCMP_ACT_KILL`. `PR_SET_NO_NEW_PRIVS` is set before
`seccomp_load` — do not reorder.

Syscalls for argument-gated features (e.g., `claims_env=`,
`rhost_policy=kernel`) enter the allowlist unconditionally: the sandbox
is loaded once, before argument-driven code paths diverge, and there is
no mechanism to re-evaluate filters mid-session. Enabling such a
feature therefore slightly broadens the allowlist for every session,
not only sessions that exercise the feature.

Unconditional syscalls for always-on observability paths (the
session-identity file under `/run/authnft/sessions/`) are similarly
unconditional. The write path uses `rename(2)` (or `renameat(2)` /
`renameat2(2)` depending on the host's libc dispatch) at open; the
cleanup path uses `unlink(2)` (or `unlinkat(2)`) at close. All five
variants are allowlisted because glibc and musl route to different
syscalls on different kernel / libc combinations — observed
empirically via `make trace` on Arch (bare `rename` / `unlink`) vs
Fedora (some paths reach the `*at` variants); musl and glibc < 2.28
route `rename(3)` through `renameat(AT_FDCWD, ...)`. All five are
read/write operations on a known
root-owned tmpfs directory; they do not expand the capability
surface the module already has for its primary fragment-loading
work.

The structured journal audit events (AUTHNFT_EVENT=open/close) reach
sd_journal_send, which uses the existing sd-bus socket path
(socket / sendmsg / connect / close / recvmsg) — all already in the
allowlist from the sd-bus block above. No new syscall was needed for
the audit-event channel.


Style
-----

- 4-space indent, no tabs. `.editorconfig` enforces this.
- C99. `_GNU_SOURCE` is defined via CFLAGS.
- Keep functions that cross the sandbox boundary small and auditable.
- Use `pam_syslog` for log output that the admin should see; use
  `DEBUG_PRINT` for developer traces (compiled out in release).
- Prefer `snprintf` with explicit bounds over string concatenation.
  Overflow returns `PAM_BUF_ERR`.
- Do not call `getgrnam`/`getpwnam` from any new signal handler — they
  are non-reentrant. Current usage is safe because the sandbox is
  active and no handlers are registered.
- New files get an SPDX `GPL-2.0-or-later` line and the copyright
  holder "Avinash H. Duduskar".


Tests
-----

- `make test` runs the symbol-whitelist check (stage 0) followed by the
  ten-stage unit suite in `tests/test_suite.c`. Stages 4 and 7 skip
  gracefully without `CAP_NET_ADMIN`; stage 2 skips under
  `AUTHNFT_AUDIT_MODE=1` because of a seccomp/valgrind conflict.
- **Container workflows are the recommended path.** Every test
  surface runs inside a booted-systemd Fedora container:
    make test-container              # unit suite, 10 stages, CAP_NET_ADMIN
    make test-integration-container  # pamtester end-to-end + valgrind
    make trace-container             # seccomp allowlist trace
  No host sudo required; no /etc/passwd, /etc/pam.d, nftables, or
  systemd mutation on the host. Requires `podman` only. Works on
  any host distro (Fedora, Arch, Debian, Ubuntu, RHEL, macOS via
  podman-machine).
- `make test-integration` and `make trace` remain available for
  developers who want to run the sudo-on-host path directly. They
  mutate host state (create transient users, /etc/pam.d entry, nft
  table) and clean up via trap. Use them when you need to inspect
  state mid-run or when no container runtime is available.
- Run `sudo make distclean` to tear down a leftover `inet authnft`
  table and the throwaway `/etc/pam.d/authnft_test` config.
- New behaviour does not require new tests — add one only if the
  behaviour is representative of a class of regression you would
  actually want caught.

The integration test creates and cleans up its own test user and
group automatically. Set `AUTHNFT_TEST_USER` to override the test
account name (default: `authnft-test`).

Stage matrix
------------

Unit suite (`tests/test_suite.c`):

  0     Exported symbols are exactly pam_sm_open_session and
        pam_sm_close_session
  1     util_is_valid_username rejects path traversal and shell
        metacharacters
  2     A syscall outside the allowlist triggers SIGSYS
  3     An allowlisted syscall (close) returns normally through the
        sandbox
  4     libnftables dry-run API accepts well-formed syntax
  5     util_get_cgroup_path rejects a non-authnft.slice PID (depth
        invariant enforcement)
  6     Compiled .so has full RELRO, canary, PIE, CFI (via checksec)
  7     Fragment load (skipped in unit tests — covered by integration
        10.2 / 10.11)
  8     util_normalize_ip accepts v4/v6 literals, strips IPv6 zone
        suffix, rejects hostnames and junk
  9     peer_lookup_tcp resolves the remote address of a localhost TCP
        pair via NETLINK_SOCK_DIAG
  10    Kernel-keyring tag round-trip: add_key → keyring_read_serial
        with full payload sanitization

Integration suite (`tests/integration_test.sh`):

  10.1  Group member denied when fragment is missing
  10.2  Group member allowed when a valid fragment exists
  10.3  Root bypasses the module entirely
  10.4  Fragment rejected when not root-owned
  10.5  Fragment rejected when world-writable
  10.6  Per-session chain/sets cleaned up at close via persisted
        session data
  10.7  close_session best-effort when no prior open state
  10.8  Multi-fragment composition via nftables `include`
  10.9  /run/authnft/sessions/ JSON file lifecycle + schema + perms
  10.10 AUTHNFT_EVENT=open/close emitted with shared correlation token
  10.11 Adversarial packet classification — cgroup match fires on
        allowed source, does not fire on disallowed source
  10.12 Class A/B socket-scope invariant — pre-scope socket does not
        match after task migration
  10.13 Cross-session isolation — user A's deny rule does not affect
        user B's traffic (per-session set boundary)
  —     No memory errors or leaks under Valgrind memcheck


CI gates
--------

Every push and pull request triggers:

  - GCC + Clang build matrix
  - cppcheck static analysis
  - CodeQL semantic analysis
  - ASan / UBSan sanitizer builds
  - libFuzzer harnesses (CIFuzz workflow, 60s budget per harness)
  - Weekly Coverity Scan
  - Nightly sustained fuzz cron (.github/workflows/fuzz-nightly.yml)
    — 30 min × 8 harnesses against the checked-in corpus

The seccomp allowlist in `src/sandbox.c` is `SCMP_ACT_KILL` default
with `PR_SET_NO_NEW_PRIVS`; see "Regenerating the seccomp allowlist"
below for the derivation procedure.


Invariant guards
----------------

Each of the seven "do not change lightly" invariants has at least one
automated guard. Regressions trip a known test, not a production
incident.

  #1 session-data persist  integration 10.6: open+close in a single
                           PAM handle must result in an empty set
  #2 two-call nft split    enforced by the comment and code structure;
                           no automated guard (libnftables semantics
                           are version-dependent)
  #3 element-before-frag   covered by integration 10.2 succeeding with
                           a fragment that references @session_v4
  #4 fragment perms        integration 10.4 (uid != 0) + 10.5
                           (world-writable) must both be rejected
  #5 seccomp provenance    `make trace` produces a syscall summary;
                           diff against src/sandbox.c before adding
  #6 open/close asymmetry  integration 10.7: close without prior open
                           must return PAM_SUCCESS
                           integration 10.14: failed open must leave
                           no nft state and no scope unit (rollback)
  #7 exported symbols      `make test-symbols` (also wired into
                           `make test`) asserts exactly
                           pam_sm_open_session + pam_sm_close_session

If you touch an invariant, run the guarding test locally and mention
it in the pull-request template checkbox.


Regenerating the seccomp allowlist
----------------------------------

Two targets cover different code paths:

    make trace

Runs the integration harness with the sandbox bypassed under
`strace -f -c`, writing a syscall-usage summary to `trace.log`.
Authoritative for the default open_session + close_session path.
Requires root and modifies system state (creates a test user and PAM
config; tears them down on exit).

    make trace-features

Runs the unit suite under `strace -f -c`, writing to
`trace-features.log`. Exercises code paths that the pamtester-driven
trace does not reach: `rhost_policy=kernel` sock_diag (unit stage 9)
and `claims_env` keyctl reads (unit stage 10). Root-free and
side-effect-free. Use this when adding a feature whose syscall
footprint cannot easily be reached from a plain pamtester run.

The unit-suite trace picks up syscalls issued by the test harness
itself (add_key as a stage 10 fixture, listen/accept for the stage 9
TCP pair, wait4/clone for stages 2 and 3, personality as the
deliberately forbidden stage 2 syscall). The trace_features.sh output
lists these explicitly; subtract them before comparing against
`src/sandbox.c`.

In both cases: compare against the allowlist in `src/sandbox.c` and
add any newly observed production syscall with a one-line
justification comment. Do not add syscalls without strace evidence.


Future directions
-----------------

The natural improvement path is BPF cgroup programs
(`BPF_CGROUP_INET_INGRESS` / `BPF_CGROUP_INET_EGRESS`), which
attach directly to a cgroup and eliminate the Class A/B socket-scope
limitation, the SYN classification gap, and the container namespace
mismatch that the current `socket cgroupv2` approach has. Not planned
for the 0.x series. See docs/TODO.txt for details.


Submitting changes
------------------

- Keep pull requests focused. Unrelated cleanups belong in separate PRs.
- Mention which invariant from the list above your change touches, if
  any, and why the change is safe.
- Run `make clean && make && make test` before submitting.
- If your change reaches a new syscall, include the strace excerpt that
  justifies the allowlist addition.
- Signed-off-by is not required, but clear commit messages are.
