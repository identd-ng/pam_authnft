pam_authnft architecture
========================

Goal
----

Bind an nftables rule to an authenticated user session for the lifetime
of that session, without a dedicated login shell, setuid binary, or
kernel module. OpenBSD's pf achieves this with anchors and pfctl.
pam_authnft achieves it on Linux with a named nftables set keyed on
`(cgroup_path, src_ip)`, plus a systemd transient scope that pins the
session's processes to a stable cgroup.


Moving parts
------------

    PAM service (sshd, login, ...)
        |
        | open_session
        v
    +------------------------------+
    |  pam_authnft.so              |
    |  (runs in the PAM process)   |
    +------------------------------+
        |    |              |
        |    |              +---> libnftables ---> kernel nf_tables
        |    |
        |    +---> sd-bus StartTransientUnit ---> systemd ---> cgroupv2
        |
        +---> libseccomp ---> kernel seccomp-BPF (filters self)


Session identity
----------------

systemd, asked over D-Bus to start a transient `.scope` with
`PIDs=[session_pid]`, places the session process into a fresh cgroup
under `authnft.slice`. The module constructs the cgroup path
deterministically from the scope unit name (e.g.,
`authnft.slice/authnft-alice-12345.scope`) and passes it as a quoted
string to nftables. The kernel resolves the path to a cgroupv2 inode
at rule-load time via the `socket cgroupv2 level 2` expression; no
`stat(2)` is performed by the module.

At packet classification time, `socket cgroupv2 level 2` in nftables
reads the cgroupv2 inode from the socket's originating cgroup. Each
session gets its own chain and three per-session sets
(`session_<user>_<pid>_v4`, `_v6`, `_cg`); a jump rule in the shared
`filter` chain dispatches to the per-session chain. This is the
pf-anchor-equivalent isolation model: one session's rules cannot
match another session's traffic.

The per-session element goes into one of the three sets based on the
resolved IP family. The cgroup-only set (`_cg`) is used when
`PAM_RHOST` is absent or not a parseable IP literal — most commonly
when sshd is configured with `UseDNS yes` and hands PAM a hostname.
Under the default `rhost_policy=lax`, the session is still pinned by
cgroup; only the src_ip leg of the tuple is dropped.
`rhost_policy=strict` restores the historical hard deny.

An optional `claims_env=NAME` argument tells the module to look up the
named PAM environment variable for a kernel-keyring serial. If set, the
keyed payload is read via `keyctl(2)` and sanitized into the
nftables element comment, providing a forensic correlation back to
whatever root-trusted PAM module produced the entry. The keyring entry
is created and owned by that producer; pam_authnft is read-only on this
path. See `keyctl(2)` and `keyrings(7)`.

`rhost_policy=kernel` is the strongest option: the module derives the
peer IP from the session process's own ESTABLISHED TCP socket via
NETLINK_SOCK_DIAG (see `ss(8)`), querying AF_INET6 first then AF_INET.
The result is normalized through `util_normalize_ip` (including
v4-mapped v6 extraction) and the module logs a warning when the
normalized kernel-reported address diverges from `PAM_RHOST`. This catches
PROXY-protocol / load-balancer misconfigurations and `UseDNS`
hostname aliasing for free, and removes the PAM caller from the trust
path for the src_ip binding.


Open-session sequence
---------------------

    1. PAM_USER validated (util_is_valid_username)
    2. Root short-circuits to PAM_SUCCESS
    3. PAM_RHOST retrieved and normalized (util_normalize_ip): IPv4/IPv6
       literals accepted; IPv6 zone suffix "%zone" stripped; v4-mapped
       v6 addresses (::ffff:a.b.c.d) extracted to plain IPv4; hostnames
       and unparseable values fall through (lax) or deny (strict)
    4. sandbox_apply installs SCMP_ACT_KILL filter (PR_SET_NO_NEW_PRIVS
       set first)
    5. bus_handler_start invokes StartTransientUnit:
         unit = "authnft-<user>-<pid>.scope"
         slice = "authnft.slice"
         job mode = "fail"
    6. cg_path constructed deterministically from scope_unit
       ("authnft.slice/authnft-<user>-<pid>.scope")
    7. cg_path + scope_unit + per-session chain/set names + remote_ip
       persisted via pam_set_data("authnft_cg_id", authnft_session_t)
       (the free callback is registered so PAM reclaims it)
    8. nft_handler_setup:
         - getgrnam("authnft") + getpwnam(user); non-members pass through
         - stat(fragment): st_uid == 0 and !(mode & S_IWOTH)
         - content validation: verb scan (reject flush/delete/reset/
           list/rename), include-path check (absolute, under
           /etc/authnft/, no globs)
         - call 1: add table + shared filter chain + ct state rule +
                   per-session chain + 3 per-session sets + element
         - call 2 (ECHO|HANDLE): add jump rule in shared filter chain;
                   parse kernel-assigned handle from output buffer
         - call 3: read fragment, substitute four placeholders
                   (@session_v4/v6/cg/chain → live names), execute

Placeholder substitution is token-aware (skips #comments and "..."
strings) and covers the top-level fragment only. Files pulled in
via nftables `include` are parsed by libnftables directly and do
not receive substitution. This is a design choice: per-session
rules use placeholders; shared includes use the shared filter chain.

The fragment in call 3 may contain `include` directives; libnftables
resolves these transitively at parse time. pam_authnft performs
ownership and mode checks only on the top-level fragment at
/etc/authnft/users/<user>. See docs/INTEGRATIONS.txt §4.6 for the
composition contract, security obligations on included files, and
cycle-detection guidance.


Close-session sequence
----------------------

    1. Root short-circuits
    2. Session data (chain name, set names, jump handle) retrieved
       from PAM data
    3. nft_handler_cleanup: delete jump rule by handle, flush and
       delete per-session chain, delete three per-session sets
       (single transaction)
    4. /run/authnft/sessions/<scope_unit>.json unlinked
    5. event_close_emit: AUTHNFT_EVENT=close with the open-side
       correlation token (see "Audit events" below)

Cleanup failures log a warning and return PAM_SUCCESS.
Session-file orphans are reaped by systemd-tmpfiles (MaxAge=7d)
per data/authnft.tmpfiles.


Failed-open rollback
--------------------

When open_session fails after partial state has been created,
pam_authnft rolls back to a clean slate before returning the
PAM error. There are two layers:

    1. nft_handler_setup runs an internal partial-cleanup on every
       error path that fires after call 1 succeeded. It removes the
       per-session chain, the three per-session sets, and (when
       call 2 had captured a handle) the jump rule. A best-effort
       single nft transaction; tolerates absent objects.
    2. pam_sm_open_session calls bus_handler_stop on every error
       path that fires after bus_handler_start succeeded. StopUnit
       on the transient scope; tolerates "unit not found" so a
       race where systemd reaped the scope already is benign.

Net result: a failed open leaves the kernel and systemd state
indistinguishable from "session was never attempted." A caller may
retry without resource accumulation.

Documented edge case: if call 2 commits the jump rule but the
echo-format handle parse fails, the rule leaks because nftables
has no delete-by-name primitive. Empirically rare (requires a
libnftables regression). Integration test 10.14 guards the common
path; a regression would either leave per-session chains/sets in
the table or leave authnft-*.scope units active.


Session-identity files
----------------------

After open_session succeeds, pam_authnft writes
/run/authnft/sessions/<scope_unit>.json with the same identity data
held in PAM data. The write uses a tempfile + rename(2) pattern so
readers never see a partial file. Unprivileged observers (SIEM
agents, workload schedulers, monitoring daemons, operator consoles)
read these files to correlate a session scope back to a pam_authnft
session without needing a PAM handle or root access to module
source. Schema and lifecycle are documented in
docs/INTEGRATIONS.txt §5.6.


Audit events
------------

Two audit channels run in parallel:

  1. systemd journal (structured fields).
     Every open_session success emits AUTHNFT_EVENT=open via
     sd_journal_send(3), and every close_session that ran against a
     valid session emits AUTHNFT_EVENT=close. Both carry an
     AUTHNFT_CORRELATION token — either inherited from PAM env
     "AUTHNFT_CORRELATION" (set by an upstream identity broker) or
     synthesized from clock_gettime+getpid+getrandom at open — and
     reused at close via authnft_session_t. SIEM consumers join the
     two events on that token.

  2. Linux audit subsystem (audit(7) AUDIT_USER_ERR).
     Every fragment-rejection path emits a parallel record via
     libaudit's audit_log_user_message(). The application body of
     the audit record carries op=authnft-fragment-reject, a short
     reason tag (missing | perms | content | nft-syntax), the PAM
     user, and the fragment path. SOC consumers that scrape
     /var/log/audit/audit.log instead of /var/log/auth.log receive
     denied-session events without subscribing to the systemd
     journal. Audit-subsystem failure is non-fatal; the
     PAM_AUTH_ERR rejection still propagates.

Contract, fields, and fallback behaviour:
  journal events       — docs/INTEGRATIONS.txt §6.2
  audit-syscall events — docs/INTEGRATIONS.txt §6.2.7


Trust model
-----------

- The module runs with whatever privilege PAM grants the calling service
  (typically root). There is no privilege drop; instead, the attack
  surface is contained with seccomp-BPF.
- Fragments must be root-owned and not world-writable. This is the same
  model used by /etc/nftables.conf and sudoers includes. Administrators
  author fragments; users never do.
- Non-members of the `authnft` group pass through without nftables
  interaction. Membership is the opt-in.
- Usernames are validated against a conservative charset
  (`[A-Za-z0-9._-]`, max 32, no leading `-` or `.`) before being
  interpolated into nft commands, D-Bus unit names, or filesystem paths.
- Socket-scope inheritance is alloc-time, not migrate-time. The kernel
  sets `sk->sk_cgrp_data` when a socket is created (`sk_alloc` →
  `cgroup_sk_alloc`) and never updates it when the owning task moves
  between cgroups. This creates two classes of session socket:
    - **Class A (in-scope):** sockets created after `bus_handler_start`
      has moved the PAM process into `authnft.slice/<scope>.scope`.
      Listeners the user opens, outbound connections from their shell.
      `socket cgroupv2 level 2` correctly returns the session scope's
      inode. The match fires.
    - **Class B (pre-scope):** sockets that existed before
      `bus_handler_start` ran — including the SSH TCP connection
      itself (accepted by sshd, inherited by the privsep child, all
      before the PAM session stack runs). These carry sshd's cgroup
      (typically `system.slice`). `socket cgroupv2 level 2` returns
      that cgroup, NOT the session scope. The match does not fire.
  Class B traffic survives because pam_authnft adds
  `ct state established,related accept` to the shared `filter`
  chain ahead of any jump rules. This short-circuits before
  the cgroup match. Integration test 10.12 validates this
  invariant empirically.


Seccomp
-------

`SCMP_ACT_KILL` default. The allowlist is strace-derived: only syscalls
observed after `sandbox_apply()` returns during a full open+close cycle
are included. PR_SET_NO_NEW_PRIVS is set before `seccomp_load`. The
bypass (env `AUTHNFT_NO_SANDBOX` or PAM arg `AUTHNFT_NO_SANDBOX=1`) is
for developer iteration only; do not enable it in deployment.


What does not live here
-----------------------

- Authentication: pam_authnft is a session module, not an auth module.
  `PAM_USER` is accepted from whatever auth stack ran before it.
- Fragment content: structural and content validation happens
  (ownership, mode, verb scan, include-path check, syntactic check
  by libnftables). Semantic choices
  — which ports to open, whether to NAT, etc. — are the administrator's.
- Per-session resource limits: provided by systemd via the transient
  scope and the parent `authnft.slice`. See `man
  systemd.resource-control(5)` and `data/authnft.slice`.


Known limits
------------

- cgroupv2 unified hierarchy only. Hybrid mounts are untested.
- Hard systemd dependency for the transient scope.
- On PAM stacks where the process invoking open_session is not the
  direct parent of the user session (forking daemon that hands off
  before the module runs), the resolved cgroup may differ from the
  session's eventual cgroup.
- Fragment semantic errors pass validation; administrators own them.
- Fragment content validation rejects flush/delete/reset/list/rename
  verbs and include paths outside /etc/authnft/. This is defence in
  depth, not a primary trust boundary (fragments are root-owned).
- Placeholder substitution (@session_v4, @session_v6, @session_cg,
  @session_chain) is applied to the top-level fragment only. Files
  pulled in via `include` do not receive substitution. Hand-written
  fragments that use a placeholder inside an included file will fail
  to load (detectable error, not silent).
  Integration test 10.13 verifies cross-session isolation.
